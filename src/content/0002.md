난 알고리즘을 잘 모른다. 애초에 알고리즘 공부에 시간을 투자하지도 않았다. 그런데 지금 생각해보면 잘한 선택인 것 같기도 하다. 그 이유는 몇 가지가 있다.

* 나는 주로 클라이언트 웹앱 코딩을 좋아하는데, 적어도 내가 만드는 수준에서는 퍼포먼스를 크게 신경 쓸 일이 없다.
* 문제가 생긴다 해도 내 알고리즘 문제가 아니라 보통 UI 렌더링 문제다.

  * 불필요한 리액트 컴포넌트 리렌더링
  * 웹페이지는 싱글 스레드로 돌아가기 때문에 약간의 계산만 있어도 어느 정도 버벅거릴 수 있음

    * 그래서 리액트에서 제공하는 여러 hook을 사용해 브라우저 렌더링을 막지 않도록 여유를 주거나
    * setTimeout 또는 requestAnimationFrame을 사용해야 한다

      * 정작 이런 방법들도 실제로는 쓸 일이 잘 없다
* 그래도 구현 과정에서는 자료구조나 알고리즘 지식이 필요하다.
* 체감상 트리 자료구조가 가장 많이 쓰이는 것 같은데

  * 몸을 비틀며 구현하다 보면 자연스럽게 익혀진다.
* 다만 최적화가 필요할 수도 있는 경우가 있는데 예를 들면 Mdit으로 연 폴더에 파일이 10000개 있는 상황이다.
* 이런 일이 없길 바라지만 생긴다면 내 능력으로는 한계가 있다.
* 그런데 AI가 있다는 사실. AI에게 최적화를 맡긴다.

  * 그렇지만 AI가 작성해준다 해도 동작 원리를 내가 이해해야 마음이 놓인다.
  * 내 머리가 이걸 완전히 이해하지 못하더라도 마음 놓고 쓸 수 있는 방법이 있다.

### 고립시키자!!!

알고리즘이 필요한 곳은 보통 순수 함수이다.

```typescript
function buildTree(data) {
	// ...
}
```

이렇게 트리를 구성하는 함수를 작성해야 할 때 알고리즘 문제를 풀며 연습했다면 머리를 꽁꽁 싸매고 코드를 작성하면 된다.

그렇지만 난 잘 못하기도 하고 머리 쓰는 게 싫기 때문에 AI에게 명령을 내린다. `이거 해줘`

```typescript
function buildTree(data) {
	// aalkdsfklmcmlxzxvcv
	// adsfczxccxvxcv
	// 어려운 코드들
}
```

이해해보려고 하지만 이해가 되지 않는다. 그래서 `이거 확실해?`라고 다시 물어봤자 무슨 소용이 있겠어.

그래서 테스트 코드를 작성하게 된다.

```typescript
it('이거 맞지?', () => {
	const tree = buildTree(data)
	expect(tree).toBe('내가 기대한 값')
})
```

테스트 코드를 정상적으로 통과했다. 근데 뭔가 불안해. 그래서 몇 가지 케이스를 더 추가한다.

```typescript
it('이거 맞지? 2', () => {
	const tree = buildTree(data2)
	expect(tree).toBe('내가 기대한 값 2')
})
it('이거 맞지? 3', () => {
	const tree = buildTree(data3)
	expect(tree).toBe('내가 기대한 값 3')
})
```

다 통과했다. 조금은 안심이 된다. 그래도 혹시 모르니까 AI에게 엣지 케이스를 위한 테스트를 작성해달라고 한다.

```typescript
it('AI가 작성한 이거 맞지?', () => {
	const tree = buildTree(data)
	expect(tree).toBe('AI가 기대한 값')
})
```

그런데 AI가 작성한 테스트가 믿음이 가지 않는다. 그래서 직접 테스트 코드를 보면서 이해한다. ㅇㅋ 만족

그래도 테스트 코드는 내가 이해할 수 있는 수준이기 때문에 buildTree 내부를 이해하지 못하더라도 잘 동작한다고 믿게 됐다.

### 신뢰

결국 그냥 신뢰의 문제이다. 내가 작성한 코드를 내가 이해하고 문제 없다 믿고 그대로 가는 것과 테스트 케이스를 꼼꼼히 작성해서 검증하는 것은 근본적으로 결함이 다 있고 믿음의 정도 차이이다.

물론 믿음의 정도를 높이기 위해 테스트 코드를 작성하는 것뿐 아니라 코드 변경이 있을 때 이걸 잡아주는 역할도 엄청 크다. 근데 이것도 마찬가지로 본인이 코드에 자신있으면 무슨 상관이겠는가.

그렇지만 팀이나 오픈소스 프로젝트의 경우, 다양한 사람들이 코드를 기여하기 때문에 예기치 않은 오류 발생 가능성이 높아지므로 테스트 코드를 꼼꼼하게 작성하는 경향이 있다.

### 결론

알고리즘을 잘 모르는 내가 AI를 잘 활용하려면 내가 이해하고 있는 부분과 이해하지 못하는 부분을 명확히 구분하고 테스트 코드로 이를 믿을 수 있게 만들어야 한다.

근데 바이브 코딩의 경우에는 앱 전체를 이해하지 못하는 영역이 되어버려서 신뢰하기 힘들다는 것이 현재 내가 느끼는 바이브 코딩의 문제점이다. 하지만 서버가 없거나 유저 데이터를 다루지 않는 클라이언트 앱의 경우에는 큰 상관없다고 생각한다. 어차피 가장 최악의 상황은 앱이 크래쉬나서 종료되는 상황이기 때문이니까.
